# Contexto del proyecto: Pridethink

App social para la comunidad LGBTQ+ centrada en identidad libre: los usuarios
escriben o eligen su propia etiqueta de identidad en vez de elegir de
categorías fijas. Prototipo funcional en fase de prueba con un socio.

## Stack técnico

- **Frontend:** React (Create React App)
- **Backend:** Firebase Auth (correo/contraseña) + Firestore (base de datos en tiempo real)
- **Hosting:** Vercel, plan Hobby (gratis) — desplegado en `https://pridethink.vercel.app`
- **Repositorio:** GitHub, `pridethink2026-ux/pridethink`, rama `main`
- **Carpeta local:** `C:\Users\HP\pridethink`

## Estructura de archivos (todo en `src/`)

- `firebase.js` — configuración e inicialización de Firebase (auth, db exportados)
- `translations.js` — diccionario de textos FIJOS de la interfaz en español e inglés (ver el punto 30 en "Funcionalidades ya construidas" para cómo agregar un texto nuevo)
- `LanguageContext.jsx` — contexto de React (`LanguageProvider` / `useLanguage()`) que expone el idioma activo de la interfaz y la función `t()` para traducir, en toda la app (ver el punto 30)
- `themes.js` — sistema de temas visuales: define las paletas (variables CSS) y las funciones para aplicar/guardar/leer el tema y calcular el tema "Rotativo". También define `--font-display` (Sora) y `--font-body` (Inter), aplicadas siempre sin importar el tema de colores activo (ver "Rediseño visual" más abajo)
- `utils.js` — helpers compartidos: `useIsMobile`, `notify()` (crear notificaciones), `timeAgo()` (tiempo relativo en español), `extractHashtags()` / `splitTextWithHashtags()` (hashtags)
- `Avatar.jsx` — círculo con iniciales sobre gradiente de color determinista por uid (mismo usuario = mismo color siempre), con un anillo alrededor que refleja la paleta pride de la identidad libre del usuario (ver `identityStyles.js` abajo, y el punto 28 en "Funcionalidades ya construidas"). Tamaños `sm`/`md`/`lg`. Se usa en el muro, comentarios, chat, notificaciones, búsqueda y perfil
- `App.js` — navegación principal: barra superior en escritorio (Perfil / Muro / Chat / Buscar + botón 🎨 de temas + campanita), barra inferior fija en móvil (Muro / Buscar / Chat / Avisos / Perfil), y el enrutamiento a las vistas superpuestas: perfiles públicos (`UserProfile`), "Guardados" (`SavedPosts`) y una publicación individual (`PostView`) — ver el punto 34 en "Funcionalidades ya construidas". La pestaña "Muro" (Inicio) usa `HomeNavIcon.jsx` en vez de un ícono fijo (ver "Ícono de Inicio personalizado" más abajo)
- `identityStyles.js` — lógica pura (sin JSX) COMPARTIDA que detecta si el texto libre de identidad de un usuario matchea una bandera pride conocida: `getIdentityColors(identityText)` devuelve un arreglo de colores hex (2 a 5 tonos) o `null` si no matchea nada. También exporta `normalizeIdentityText` (quita acentos/mayúsculas para que la detección de palabras clave no dependa de cómo se escribió el texto). La usan tanto `homeIcon.js` (para el ícono de Inicio) como `Avatar.jsx` (para el anillo del avatar) — un solo lugar para agregar una bandera nueva (`COLOR_RULES` en este archivo) y ambos componentes la usan al instante
- `profileFields.js` — lógica pura (sin JSX) COMPARTIDA sobre los datos personales del registro: lista de países (`getCountryOptions(locale)` / `getCountryName(code, locale)`), idiomas soportados (`LANGUAGE_OPTIONS`), opciones de género con etiqueta traducida (`getGenderOptions(t)`) y cálculo de edad exacta (`calculateAge(birthDateLike)`, acepta string `"YYYY-MM-DD"` o un `Date`). La usan `AuthProfile.jsx` (paso 2 del registro) y `ProfileAbout.jsx` (sección "Acerca de" del perfil) — ver el punto 31 en "Funcionalidades ya construidas"
- `ProfileAbout.jsx` — componente compartido de la sección "Acerca de" del perfil (nombre completo, género, país, idioma, fecha de nacimiento con edad calculada al vuelo). Lo usan tanto `AuthProfile.jsx` (tu propio perfil) como `UserProfile.jsx` (perfil público de otra persona) — ver el punto 31
- `homeIcon.js` — lógica pura (sin JSX) que decide qué FORMA le toca al ícono de Inicio según el texto de identidad del usuario (`SHAPE_RULES`, exclusivo de este ícono); para el color reutiliza `getIdentityColors` de `identityStyles.js` en vez de duplicarlo. `getHomeIcon(identityText)` devuelve `{ shape, colors }`
- `HomeNavIcon.jsx` — componente que dibuja el SVG del ícono de Inicio usando lo que devuelve `getHomeIcon`. Se llama `HomeNavIcon.jsx` (no `HomeIcon.jsx`) a propósito: Windows no distingue mayúsculas/minúsculas en nombres de archivo, así que un componente `HomeIcon.jsx` chocaría con `homeIcon.js` (webpack terminaba resolviendo el import al archivo equivocado). Si en el futuro se agrega otro archivo cuyo nombre solo se diferencie de uno existente por mayúsculas/minúsculas, hay que elegir un nombre distinto de verdad, no solo un cambio de capitalización
- `AuthProfile.jsx` — login/registro (en pasos: Cuenta → Datos personales → Identidad, con verificación de mayoría de edad — ver el punto 29 en "Funcionalidades ya construidas"), TU perfil (identidad, sección "Acerca de", privacidad de perfil y de muro, contadores de seguidores/seguidos)
- `UserProfile.jsx` — perfil PÚBLICO de cualquier usuario (avatar grande, sección "Acerca de", contadores de seguidores/seguidos, sus publicaciones, botón seguir, botón Reportar)
- `FollowListModal.jsx` — modal reutilizable que muestra la lista de "Seguidores" o "Siguiendo" de un usuario (avatar + nombre, click para ir a su perfil). Lo usan tanto `UserProfile.jsx` como `AuthProfile.jsx`
- `ReportButton.jsx` — botón "⋮" + menú "Reportar" + modal para reportar una publicación o un usuario (motivo + detalle opcional). Reutilizable: lo usan `Feed.jsx` (`PostCard`, reporta un post) y `UserProfile.jsx` (reporta al usuario del perfil) — ver el punto 32 en "Funcionalidades ya construidas"
- `Reactions.jsx` — lógica y UI COMPARTIDA de reacciones (5 tipos, mapa `{ uid: tipo }`, hook `useReactionPicker()` para abrir el selector con hover/long-press, componente `ReactionPicker`). La usan `Feed.jsx` (reacciones a un post) y `Chat.jsx` (reacciones a un mensaje) — ver el punto 33 en "Funcionalidades ya construidas"
- `SavedPosts.jsx` — pantalla "Guardados", accesible desde tu perfil (`AuthProfile.jsx` → botón "Ver guardados"). Lista tus publicaciones guardadas reutilizando `PostCard` — ver el punto 34
- `PostView.jsx` — publicación individual ("post original"), a donde navega tocar la vista previa de un post compartido en el chat. Reutiliza `PostCard` — ver el punto 34
- `SharePostModal.jsx` — selector de contactos para "Compartir por chat" un post (importa `getChatId` de `Chat.jsx`). Lo abre el botón 📤 de `PostCard` — ver el punto 34
- `Mentions.jsx` — lógica y UI COMPARTIDA de menciones `@usuario` en posts y comentarios: hook `useAllUsers()` (escucha toda la colección `users`, mismo patrón que ya usaban `Feed.jsx`/`Search.jsx`), `useMentionAutocomplete()` (detecta el "@palabra" en curso mientras se escribe y arma sugerencias), componente `MentionSuggestions` (el menú desplegable), `parseMentions()`/`extractMentionedUids()` (detectan menciones reales comparando contra los nombres de usuarios existentes) y `renderTextWithMentions()` (pinta cada mención detectada como texto clickeable). La usa `Feed.jsx` en el composer, en los comentarios y al editar un post — ver el punto 35
- `Feed.jsx` — muro social (posts, reacciones, comentarios, editar/borrar, hashtags, pestañas Todos/Siguiendo, botón Reportar en publicaciones ajenas, botones Guardar/Compartir). Exporta también `PostCard`, reutilizado por `UserProfile.jsx`, `SavedPosts.jsx` y `PostView.jsx`
- `Search.jsx` — buscador de personas y publicaciones
- `Chat.jsx` — mensajería en tiempo real entre usuarios, con notas de voz (ícono de micrófono SVG inline, ver "Rediseño visual" más abajo), reacciones a mensajes (ver el punto 33) y mensajes de "post compartido" (ver el punto 34). Exporta `getChatId`, que también usa `SharePostModal.jsx`
- `Notifications.jsx` — lógica y UI de notificaciones: hook `useNotifications`, campanita de escritorio (default export) y `NotificationsScreen` (pantalla completa para la barra inferior en móvil)
- `public/logo-icon.png`, `logo192.png`, `logo512.png`, `favicon.ico` — logo de la marca
- `public/index.html` — además del HTML base, carga las Google Fonts "Sora" e "Inter" (ver "Rediseño visual" más abajo)
- `firestore.rules` (en la raíz del proyecto, no en `src/`) — reglas de seguridad de Firestore, listas para pegar en Firebase console. Ver sección "Seguridad de Firestore" más abajo.

## Estructura de datos en Firestore

- `users/{uid}` → `{ email, displayName, identity, joinedAt, identityUpdatedAt, isPrivate, isWallPrivate, blockedUsers: [uid...], following: [uid...], fullName, birthDate, country, language, gender, genderOther }`
  - `fullName`, `birthDate`, `country`, `language`, `gender`, `genderOther` se capturan en el registro por pasos — ver el punto 29 en "Funcionalidades ya construidas" para el detalle completo de cada uno y de la verificación de edad.
  - `isWallPrivate` es nuevo (booleano, default `false`): oculta SOLO tus publicaciones cuando alguien visita tu perfil público, sin ocultar el resto de tu información. Es independiente de `isPrivate` — ver el punto 31 para el detalle completo de ambos.
  - `blockedUsers` es de UNA sola dirección: solo vive en el documento de quien bloquea (nunca se escribe nada en el documento de la persona bloqueada). Bloquear/desbloquear siempre es `arrayUnion`/`arrayRemove` sobre tu propio documento. La app oculta a alguien si CUALQUIERA de los dos bloqueó al otro (revisando ambos `blockedUsers` al filtrar), pero solo quien hizo el bloqueo puede deshacerlo — por eso "Desbloquear" solo aparece cuando el bloqueo fue tuyo.
  - `following` es nuevo: la lista de uids que ESTE usuario sigue (mismo patrón que `blockedUsers`, con `arrayUnion`/`arrayRemove`). Los seguidores de alguien NO se guardan aparte: se derivan en tiempo real con una consulta `where("following", "array-contains", uid)` sobre toda la colección `users`.
- `posts/{postId}` → `{ authorId, authorName, authorIdentity, text, createdAt, reactions: { [uid]: tipo }, hashtags: ["..."], mentionedUids: [uid...] }`
  - `hashtags` es nuevo: se detectan automáticamente del texto al publicar o editar (`utils.extractHashtags`), guardados en minúsculas y sin duplicados.
  - `reactions` reemplazó a lo que antes era `likes: [uid...]` — ver el punto 33 en "Funcionalidades ya construidas" para el detalle completo (los 5 tipos, cómo se cuenta, y qué pasa con los `likes` de posts viejos).
  - `mentionedUids` es nuevo (punto 35): los uids de las personas mencionadas con `@nombre` en el texto, detectados con `Mentions.extractMentionedUids`. Solo se usa para saber a quién notificar (y, al editar, a quién notificar de nuevo si la mención es nueva) — el renderizado de las menciones NO depende de este campo, se vuelve a calcular comparando el texto contra los usuarios reales cada vez que se muestra.
- `posts/{postId}/comments/{commentId}` → `{ authorId, authorName, authorIdentity, text, createdAt, mentionedUids: [uid...] }`
  - El conteo de comentarios NO se guarda como campo aparte: se escucha la subcolección en tiempo real y se cuenta directo, para que nunca se desincronice. Lo mismo aplica a seguidores (contados en vivo) y a la pestaña "Siguiendo" del muro (filtrada en el momento, no guardada).
  - `mentionedUids` (punto 35): mismo propósito que en `posts/{postId}`, para notificar menciones dentro de un comentario.
- `chats/{chatId}/messages/{msgId}` → mensaje de texto: `{ senderId, text, createdAt, reactions: { [uid]: tipo } }`; mensaje de nota de voz: `{ senderId, type: "audio", audioData, audioDuration, createdAt, reactions: { [uid]: tipo } }`; mensaje de post compartido: `{ senderId, type: "shared_post", sharedPostId, createdAt, reactions: { [uid]: tipo } }` (chatId = los dos UID ordenados y unidos con "_")
  - `audioData` es el audio codificado en base64 (formato webm/opus, ~32kbps) guardado directo en el documento — no se usa Firebase Storage. Con el límite de 60 segundos de grabación pesa ~300KB como máximo, bien por debajo del límite de 1MB por documento de Firestore.
  - `reactions` es nuevo (punto 33): igual formato y mismos 5 tipos que en `posts`, pero acá como mucho reaccionan 2 personas (los participantes del chat).
  - `type: "shared_post"` es nuevo (punto 34): solo guarda `sharedPostId`, una referencia — NUNCA una copia del autor/texto del post. La vista previa que se ve en el chat se resuelve en vivo leyendo `posts/{sharedPostId}`. Ver el punto 34 para el detalle completo.
- `notifications/{uid}/items/{itemId}` → `{ type: 'like'|'comment'|'message'|'follow'|'mention', fromUid, fromName, fromIdentity, createdAt, read, postId? }`
  - `'follow'` es nuevo: se crea cuando alguien te empieza a seguir.
  - `'mention'` es nuevo (punto 35): se crea cuando alguien te menciona con `@nombre` en un post o comentario. Es el único tipo que trae `postId` (el post donde ocurrió la mención) — al tocar esa notificación se navega directo a `PostView` en vez de al perfil de quien la generó, a diferencia del resto de los tipos.
- `reports/{reportId}` → `{ reporterUid, targetType: 'post'|'user', targetId, reason, details, createdAt, status: 'pending' }` — nueva, para moderación. Ver el punto 32 en "Funcionalidades ya construidas" y la sección "Seguridad de Firestore" para el detalle completo (incluyendo por qué el id del documento NO es autogenerado).
- `savedPosts/{uid}/items/{postId}` → `{ savedAt }` — nueva (punto 34), lista personal de publicaciones guardadas. El id del documento ES el id del post (no autogenerado): guardar es `setDoc` con ese id, quitar es `deleteDoc` sobre el mismo id, y "¿está guardado?" es solo "¿existe el documento?". No guarda ninguna copia del contenido del post, solo la referencia.

## Funcionalidades ya construidas

1. Registro/login real con Firebase Auth
2. Sesión persistente (no vuelve al login al recargar)
3. Cambiar identidad libremente cuando quieras (con historial de última actualización)
4. Perfil privado (oculta tus posts y tu presencia en el chat de otros usuarios; en tu perfil público solo se ve un aviso, sin publicaciones)
5. Muro social: publicar, dar like (❤️ con animación de "latido"), comentar, editar y borrar tus propios posts
6. Chat en tiempo real entre usuarios, con buscador y bloqueo mutuo de usuarios
7. Diseño responsivo del chat: una columna a la vez en móvil (con botón "volver"), lado a lado en escritorio
8. Notificaciones en tiempo real (like, comentario, mensaje, seguidor nuevo) con campanita/pantalla y contador de no leídas
9. Logo de la marca en la pantalla de login y en el encabezado de navegación (responsivo, sin superponerse en móvil)
10. Temas visuales rotativos: botón 🎨 junto a la campanita abre un menú para elegir entre 4 temas — "Noche Violeta 🌙" (el original), "Arcoíris 🌈", "Océano 🌊" y "Atardecer 🌅" — o el modo "Rotativo 🔄", que cambia de tema automáticamente según el día del año. La preferencia se guarda en `localStorage` y se aplica al cargar la app seteando variables CSS (`--bg`, `--surface`, `--accent`, etc.) en `document.documentElement`, así que toda la app cambia de color al instante sin recargar.
11. Notas de voz en el chat: botón 🎤 junto al campo de texto graba audio (MediaRecorder, webm/opus, ~32kbps) con indicador de grabación y contador de tiempo, corta sola a los 60 segundos, y se puede cancelar o enviar. Se guarda en base64 dentro del propio documento del mensaje en Firestore (sin Firebase Storage). Los mensajes de audio se muestran con un reproductor simple (play/pausa + duración).
12. **Avatares con iniciales**: componente `Avatar.jsx` reutilizable — círculo con 1-2 iniciales sobre un gradiente de color determinista según el uid (mismo usuario, mismo color, siempre, en toda la app), en tamaños chico/mediano/grande.
13. **Tarjetas de publicación rediseñadas**: avatar + nombre + identidad + tiempo relativo en español ("hace 5 min", "hace 2 h", "ayer", o fecha si es más vieja).
14. **Navegación tipo app**: en móvil, barra de navegación fija inferior (Muro / Buscar / Chat / Avisos / Perfil) con puntito rojo en Avisos si hay notificaciones sin leer; en escritorio se conserva la barra superior con pestañas + campanita. Transición suave (fundido) al cambiar de vista.
15. **Skeletons de carga**: rectángulos con efecto shimmer mientras llegan las publicaciones del muro por primera vez.
16. **Perfiles públicos**: al tocar el avatar o el nombre de cualquier persona (muro, comentarios, chat, notificaciones, búsqueda) se abre su perfil — avatar grande, nombre, identidad, fecha de registro, contadores de seguidores/seguidos, y sus publicaciones (reutilizando el mismo `PostCard` del muro, con like/comentarios funcionando igual). Respeta bloqueo (en cualquier dirección, no se puede ver el perfil) y privacidad (se ve lo mínimo con un aviso). Botón "Volver" que regresa a la pestaña donde estabas.
17. **Seguir usuarios**: botón Seguir/Dejar de seguir en cualquier perfil público (nunca en el tuyo). Notifica al usuario seguido. En el muro, pestañas "Todos" / "Siguiendo" para filtrar solo publicaciones de gente que sigues.
18. **Hashtags**: se detectan automáticamente al publicar o editar un post (`#loquesea`) y se muestran como texto clickeable en color de acento; al tocar uno se filtra el muro por ese hashtag, con un chip visible para quitar el filtro.
19. **Búsqueda**: pantalla dedicada (ícono 🔍 en la navegación) que busca, entre lo ya cargado, personas por nombre y publicaciones por texto o `#hashtag`. Excluye usuarios bloqueados y perfiles privados de los resultados. Los resultados de personas abren su perfil público.
20. **Recuperar contraseña**: en la pantalla de login, enlace "¿Olvidaste tu contraseña?" (solo visible en modo login, no en registro) que lleva a una pantalla para escribir tu correo (precargado si ya lo habías escrito) y enviar un correo de restablecimiento con `sendPasswordResetEmail` de Firebase Auth. Mensajes claros en español para éxito, correo no registrado, correo mal escrito, y demasiados intentos.
21. **Reglas de seguridad de Firestore** (`firestore.rules`): reemplazan el modo de prueba. Ver la sección dedicada más abajo.
22. **Desbloquear usuarios**: antes solo se podía bloquear desde el chat, sin ninguna forma de deshacerlo. Ahora: (a) en tu perfil (`AuthProfile.jsx`) hay una sección "Usuarios bloqueados" con avatar, nombre y botón "Desbloquear" por cada persona en tu `blockedUsers` (o "No has bloqueado a nadie" si la lista está vacía); (b) si intentas ver el perfil público de alguien que TÚ bloqueaste, la pantalla "No puedes ver este perfil" incluye también un botón "Desbloquear" ahí mismo — pero si quien te bloqueó fue la OTRA persona, el mensaje se queda genérico sin botón, para no revelar quién bloqueó a quién.
23. **Ícono de micrófono rediseñado**: el botón 🎤 del chat ahora es un SVG inline (`MicIcon` en `Chat.jsx`, trazo redondeado, `currentColor`). Mientras se está grabando, el ícono se pinta con el color de acento del tema y tiene una animación de pulso suave (clase `.pt-mic-pulse` en `index.css`, combina escala y opacidad).
24. **Listas de seguidores/seguidos**: al tocar el contador "Seguidores" o "Siguiendo" (en tu perfil o en cualquier perfil público) se abre un modal (`FollowListModal.jsx`) con la lista de esas personas (avatar de inicial + nombre), calculada en tiempo real desde Firestore (igual que los contadores, nunca guardada como número aparte). Al tocar a alguien de la lista se va a su perfil público. La lista respeta bloqueos (en cualquier dirección) y oculta perfiles privados, igual que en `Search.jsx`.
25. **Rediseño visual con identidad propia**: tipografía de marca cargada desde Google Fonts (`public/index.html`) — "Sora" para títulos de sección, el logo "Pridethink" de la nav, y nombres de perfil (variable `--font-display` en `themes.js`), "Inter" para el resto del texto (`--font-body`). Estas dos variables se aplican siempre, sin importar cuál de los 4 temas de color (o el modo Rotativo) esté activo. Títulos de sección y el logo de la nav usan además un degradado de texto con los dos colores de acento del tema (`background: linear-gradient(...)` + `WebkitBackgroundClip: "text"`). Tarjetas (posts, perfil, chat, notificaciones, búsqueda) tienen bordes más redondeados y una sombra sutil (`boxShadow` con negro semitransparente, mismo patrón ya usado en los paneles desplegables); los botones principales ya usaban degradado con los colores del tema y se mantuvo.
26. **Ícono de Inicio personalizado según identidad**: la pestaña "Muro" (Inicio) de la nav —de escritorio y de la barra inferior móvil— ya no usa un ícono fijo. `homeIcon.js` (`getHomeIcon`) analiza el texto libre de identidad del usuario logueado (`users/{uid}.identity`, escuchado en tiempo real desde `App.js`, así que si editas tu identidad el ícono cambia al instante) y elige:
    - **Una forma** (SVG de solo trazo, sin relleno, dibujada en `HomeNavIcon.jsx`): casita simple por defecto; casa de perro (entrada en arco) si la identidad menciona `perro`/`perra`/`dog`/`can`; caja de arena de gato (entrada redonda) si menciona `gato`/`gata`/`cat`/`felino`/`felina`; casita de pájaro (agujero + palito) si menciona `pajaro`/`ave`/`bird`; castillo con torres y almenas si menciona `rey`/`reina`/`king`/`queen`/`princesa`/`principe`/`realeza`.
    - **Un color de trazo**: si la identidad menciona una orientación/identidad con bandera pride conocida, el trazo usa un degradado con los colores de esa bandera (colores fijos, no de tema — misma excepción que los avatares): `trans` (celeste/rosa/blanco), `no binario/a/e`/`nb`/`enby` (amarillo/blanco/morado/negro), `bi`/`bisexual` (rosa/morado/azul), `pan`/`pansexual` (rosa/amarillo/celeste), `lesbiana`/`lesbian` (degradado de naranjas a rosas), `gay`/`homosexual` (verdes a azules/morado). Si no se detecta ninguna, el trazo usa el degradado de acento del tema activo (`var(--accent)` / `var(--accent2)`), así que sigue cambiando con los 4 temas y el modo Rotativo.
    - La detección de palabras clave ignora mayúsculas/minúsculas y acentos, y usa límites de palabra para evitar falsos positivos (por ejemplo, "no binario" no dispara la regla de "bi", y "cansado" no dispara la de "can").
    - El ícono se atenúa (opacidad ~55%) cuando la pestaña Muro no está activa, y se ve a color completo cuando sí lo está — igual que el resto de la nav distingue pestaña activa/inactiva.
    - Para agregar una forma nueva alcanza con sumar una entrada a `SHAPE_RULES` en `homeIcon.js`; para una bandera nueva hay que sumarla a `COLOR_RULES` en `identityStyles.js` (compartido con el anillo del avatar, ver punto 28).
27. **Burbujas de nota de voz con waveform**: el reproductor de audio del chat (`AudioMessage` en `Chat.jsx`) ya no muestra el emoji 🎤 junto a la duración — ahora es botón play/pausa + una waveform decorativa (SVG, `AudioWaveform`) + duración.
    - La waveform son 24 barras SVG delgadas con alturas pseudo-aleatorias generadas con un PRNG determinista (`seededRandom`/`hashSeed` en `Chat.jsx`) sembrado con el `id` del mensaje en Firestore, así que cada nota de voz se ve distinta pero SIEMPRE con el mismo patrón (no cambia entre renders ni se ve diferente para quien envía vs. quien recibe).
    - Mientras el audio se reproduce, las barras se animan tipo ecualizador (clase `.pt-wave-bar.pt-wave-playing`, `@keyframes pt-wave-eq` en `index.css`, sube/baja con `scaleY`) con `animationDelay` escalonado por barra (`i * 0.045s`) para que la onda se vea viva y no sincronizada. En pausa o antes de reproducir, las barras quedan estáticas (sin la clase `pt-wave-playing`).
    - El color de las barras sale de variables de tema existentes, no de un color fijo: `var(--bg)` en burbujas propias (fondo degradado rosa del tema, necesita un trazo oscuro para contraste) y `var(--text-muted)` en burbujas del otro usuario (fondo oscuro `--surface-alt`) — mismas variables que ya usaba el texto de duración, así que el color de tema (cualquiera de los 4, o Rotativo) siempre se ve bien.

A diferencia de `UserProfile.jsx`, a propósito NO respeta `isPrivate`/`isWallPrivate` del autor — esos dos campos controlan si se puede REPASAR el muro de alguien desde su perfil, no si se puede ver un post puntual al que ya se llegó por un enlace directo (alguien te lo compartió) — coherente con que las reglas de Firestore ya permiten leer cualquier post a cualquier autenticado sin mirar esos campos. Sí respeta el BLOQUEO (en cualquier dirección), igual que en cualquier otro lado de la app, con el mismo botón "Desbloquear" si el bloqueo fue tuyo.
    - `App.js` ahora tiene tres estados de "vista superpuesta" mutuamente excluyentes (`viewingProfileUid`, `viewingSaved`, `viewingPostId`) — abrir cualquiera de las tres cierra las otras dos; ninguna pestaña de la nav se ve "activa" mientras alguna esté abierta (`anyOverlayOpen`).
    - **Reglas de seguridad nuevas** (`firestore.rules`): `savedPosts/{ownerUid}/items/{postId}` — colección totalmente privada, `allow read, create, update, delete: if esDueno(ownerUid);` (nadie más puede ver ni tocar la lista guardada de otra persona). El mensaje `type: "shared_post"` no necesitó ningún cambio de reglas: la regla de `create` de `chats/{chatId}/messages/{messageId}` ya no restringía qué campos puede tener un mensaje nuevo (solo que `senderId` sea el tuyo), así que `sharedPostId` ya quedaba permitido.
35. **Menciones `@usuario` en posts y comentarios** (`Mentions.jsx`, ver arriba): al escribir "@" seguido de letras en el composer del muro, en un comentario, o al editar tu propio post, se abre un menú desplegable con sugerencias de usuarios que coincidan por nombre o identidad, buscando en tiempo real (mismo patrón de `onSnapshot` sobre toda la colección `users` que ya usaban `Feed.jsx`/`Search.jsx`, filtrado en el cliente). El menú excluye bloqueados (en cualquier dirección) y perfiles privados, mismo criterio que ya usa `Search.jsx`.
    - **Cómo se detecta una mención (sin guardar posiciones de texto)**: al elegir a alguien del menú se inserta `@NombreCompleto ` en el texto. Tanto al publicar/comentar/editar como al RENDERIZAR, el texto se vuelve a analizar comparando cada "@algo" contra los nombres reales de usuarios existentes (`parseMentions`, buscando siempre el nombre más largo que matchee, para que "@Ana Torres" no se corte en "Ana" si hace falta el nombre completo) — así que también funciona si alguien escribe `@NombreExacto` a mano, sin pasar por el menú. Si "@algo" no matchea a ningún usuario real, se muestra tal cual, como texto normal (no rompe nada).
    - **Guardado**: al publicar un post, comentar, o editar un post, se calculan los uids mencionados (`extractMentionedUids`) y se guardan en `mentionedUids` (arreglo de uids, ver "Estructura de datos en Firestore" más arriba) — solo se usa para saber a quién notificar, nunca para el renderizado (que siempre se recalcula desde el texto).
    - **Notificación**: cada persona mencionada recibe una notificación nueva (`type: "mention"`, reutilizando `utils.notify()` y todo el sistema de notificaciones ya existente — campanita, contador de no leídas, pantalla completa en móvil) con el `postId` del post donde ocurrió la mención (sea que la mención esté en el post o en un comentario de ese post). Al tocar una notificación de mención se navega directo a `PostView` con ese post — a diferencia de like/comentario/mensaje/seguidor, que llevan al perfil de quien generó el aviso (ver `Notifications.jsx` → `NotificationItem`). Al EDITAR un post, solo se notifica a las menciones NUEVAS (comparando contra el `mentionedUids` que ya tenía guardado el post), no se vuelve a notificar a quien ya estaba mencionado desde antes — mismo principio que ya se usaba con las reacciones (no notificar de más).
    - **Renderizado**: cada "@Nombre" detectado se muestra como texto clickeable en el color de acento del tema activo (`var(--accent)`, distinto del `var(--accent2)` que ya usaban los hashtags, para poder distinguirlos a simple vista) y lleva al perfil público de esa persona al tocarlo — funciona igual en el texto de un post y en el texto de un comentario.
    - **Dónde vive la lógica de "quién puede mencionarse a quién"**: `PostCard` (en `Feed.jsx`, reutilizado por `UserProfile.jsx`/`SavedPosts.jsx`/`PostView.jsx`) trae su propia lista de usuarios con `useAllUsers()` — necesaria tanto para renderizar las menciones del post y de cada comentario, como para el autocompletado del comentario y de la edición. El composer del muro (`Feed.jsx`, la publicación nueva) hace lo mismo para su propio autocompletado.
    - **Sin cambios en `firestore.rules`**: ni la regla de `posts/{postId}` (creación/edición por el autor) ni la de `posts/{postId}/comments/{commentId}` (creación) ni la de `notifications/{uid}/items/{itemId}` (creación) restringen qué campos puede tener el documento más allá de `authorId`/`fromUid` — así que `mentionedUids` y `postId` ya quedaban permitidos sin publicar nada nuevo en la consola de Firebase.

## Seguridad de Firestore

- El archivo `firestore.rules` (raíz del proyecto) contiene las reglas de seguridad que reemplazan el modo de prueba. **Hay que pegarlas manualmente en Firebase console** (Firestore Database → Reglas → pegar todo el archivo → Publicar); esto no se sube solo con `git push`, es un paso aparte en la consola de Firebase.
- Principio general: nada es público, todo requiere sesión iniciada. Cada colección se limita a lo mínimo que la app necesita:
  - `users/{uid}`: cualquier autenticado lee cualquier perfil (lo necesitan el chat, el muro, los perfiles públicos y la búsqueda); solo el dueño crea/edita el suyo; nadie borra.
  - `posts/{postId}`: cualquier autenticado lee; solo el autor crea (firmado con su uid), edita cualquier campo, o borra; cualquier OTRA persona autenticada puede reaccionar — pero la regla solo le permite tocar el campo `reactions`, y dentro de ese mapa, solo su propia entrada (nunca la de alguien más). Ver el punto 33 y `soloTocaMiReaccion()` más abajo.
  - `posts/{postId}/comments/{commentId}`: cualquier autenticado lee y crea (firmado con su uid); solo el autor del comentario borra.
  - `chats/{chatId}` y sus `messages`: solo los dos participantes de la conversación pueden leer o escribir — se valida separando el `chatId` por "_" (así se arma en `getChatId()`) y confirmando que tu uid es una de las dos mitades. Los mensajes no se pueden editar (texto, audio, remitente) ni borrar una vez enviados, salvo una excepción: cualquiera de los dos participantes puede reaccionar (punto 33), mismo `soloTocaMiReaccion()` que en `posts`.
  - `notifications/{uid}/items/{itemId}`: solo el dueño lee/borra/marca como leídas las suyas; cualquier autenticado puede CREAR una notificación para otra persona (así funcionan los avisos de like/comentario/mensaje/seguidor), siempre firmada con su propio uid como remitente.
  - `reports/{reportId}` (nueva, punto 32): nadie lee, edita ni borra desde el cliente (`allow read, update, delete: if false;` — la única forma de verlos es entrando a la colección desde Firebase console, que usa el SDK de administrador y no pasa por estas reglas). Cualquier autenticado puede CREAR un reporte firmado con su propio uid (`reporterUid`), siempre en estado `"pending"` y con `targetType` válido (`"post"` o `"user"`). El id del documento es determinista (`${reporterUid}_${targetType}_${targetId}`, armado en `ReportButton.jsx`) en vez de autogenerado — si esa persona ya reportó ese mismo target, Firestore ve el intento como una actualización de un documento existente, no como una creación, y como acá no hay `allow update`, se rechaza solo (sin que el cliente necesite leer la colección para revisar duplicados de antemano, cosa que además las reglas no permitirían).
  - `savedPosts/{ownerUid}/items/{postId}` (nueva, punto 34): `allow read, create, update, delete: if esDueno(ownerUid);` — colección totalmente privada, cada quien solo toca su propia lista, nadie más puede leerla ni escribirla.
  - Cualquier otra colección no contemplada: denegada por defecto.
- **Desbloquear ya está cubierto sin cambios en las reglas**: tanto la sección "Usuarios bloqueados" del perfil como el botón "Desbloquear" en un perfil público hacen exactamente lo mismo que bloquear — `updateDoc` con `arrayRemove`/`arrayUnion` sobre el campo `blockedUsers` de tu propio documento (`users/{tu-uid}`) — así que ya lo permite la regla `allow create, update: if esDueno(uid);` de `users/{uid}`. No hace falta publicar nada nuevo en Firebase console para esto.
- **Los campos nuevos del registro (punto 29) tampoco necesitan cambios de reglas**: `firestore.rules` no restringe qué campos puede tener `users/{uid}` (la regla `allow create, update: if esDueno(uid);` es a nivel documento, no por campo), así que `fullName`, `birthDate`, `country`, `language`, `gender` y `genderOther` ya quedan permitidos sin tocar la consola de Firebase. Lo mismo aplica a `isWallPrivate` (punto 31): mismo `updateDoc`/`setDoc` con `merge:true` sobre tu propio documento, ya cubierto por la misma regla.
- **`soloTocaMiReaccion(antes, despues)` (punto 33)** reemplazó a la vieja `soloAgregaOQuitaMiLike`: ahora valida un MAPA (`reactions`), no un arreglo. Se reutiliza en `posts/{postId}` y en `chats/{chatId}/messages/{messageId}` — un solo lugar si algún día se agrega un sexto tipo de reacción (hay que sumarlo a la lista `['like', 'applause', 'fire', 'pride', 'funny']` dentro de esta función, además de a `REACTION_TYPES` en `Reactions.jsx`).

## Notas importantes de configuración

- Firestore ya NO depende del modo de prueba una vez publicadas las reglas de `firestore.rules` (ver arriba). Si en algún momento deja de funcionar la lectura/escritura y las reglas ya están publicadas, revisar en la consola de Firebase si algún error de permisos aparece en la pestaña de uso, o si el código está pidiendo algo que las reglas no contemplan.
- **CUIDADO con imports no usados en el código**: Vercel compila con `CI=true`, lo que convierte cualquier advertencia de ESLint (como una importación no utilizada) en un error que **rompe el despliegue por completo**, aunque en el navegador local (`npm start`) solo se vea como advertencia amarilla sin problema. Esto ya causó que 5 despliegues seguidos fallaran en silencio sin que se notara hasta revisar el dashboard de Vercel. Cualquier cambio de código debe revisarse contra esto antes de hacer `git push`.
- **Colores del código = variables CSS, no valores fijos**: ningún archivo de pantalla (`App.js`, `AuthProfile.jsx`, `UserProfile.jsx`, `Chat.jsx`, `Feed.jsx`, `Search.jsx`, `Notifications.jsx`, `FollowListModal.jsx`, `ProfileAbout.jsx`, `ReportButton.jsx`, `Reactions.jsx`, `SavedPosts.jsx`, `PostView.jsx`, `SharePostModal.jsx`, `Mentions.jsx`) usa colores hex/rgba fijos — todos usan `var(--bg)`, `var(--surface)`, `var(--accent2)`, etc. Si se agrega una pantalla o componente nuevo, debe seguir el mismo patrón para que respete el tema activo. Las paletas viven únicamente en `src/themes.js`, que además de los colores base define variantes "suaves" para fondos/bordes de mensajes: `--accent2-soft/-softer/-soft-border` (usadas para errores, acentos rosa/coral) y `--accent-soft/-softer/-soft-border` (usadas para mensajes de éxito, con el otro color de acento del tema, para que se distingan visualmente de un error). `themes.js` también define `--font-display`/`--font-body` (tipografía, no color), aplicadas siempre igual sin importar el tema.
  - **Excepciones intencionales**: `Avatar.jsx` genera el color de fondo de las iniciales con HSL a partir de un hash del uid (para que cada persona tenga SIEMPRE el mismo color de identidad, sin importar el tema activo) y usa texto blanco fijo para garantizar contraste contra cualquier tono generado — documentado en un comentario dentro del propio archivo. `identityStyles.js` (compartido por `homeIcon.js` y `Avatar.jsx`) usa colores hex fijos para las banderas pride (trans, no binario, bi, pan, lesbiana, gay): son los colores oficiales de cada bandera, no colores de identidad visual de la app, así que no deben cambiar con el tema — cuando no se detecta ninguna bandera, tanto el ícono de Inicio como el anillo del avatar caen a `var(--accent)`/`var(--accent2)` del tema activo. Las sombras (`boxShadow`) sí usan `rgba(0, 0, 0, x)` fijo en vez de una variable de tema: es una sombra neutra para dar sensación de profundidad, no un color de identidad visual, y ya era el patrón usado desde antes en los paneles desplegables (menú de temas, notificaciones).
- **Conteos derivados, nunca guardados**: comentarios por post, seguidores/seguidos por usuario (incluyendo las listas del modal `FollowListModal.jsx`, no solo el número), y la pestaña "Siguiendo" del muro se calculan siempre en tiempo real desde Firestore (subcolecciones o consultas `array-contains`/filtro en el cliente), nunca como un campo numérico aparte. Así nunca pueden desincronizarse.
- El dueño del proyecto (Rorby) prefiere comunicación en español y un enfoque guiado paso a paso; no tiene experiencia previa de programación, así que las instrucciones deben ser explícitas y sin dar por sentado conocimiento técnico.
- Flujo de publicación establecido: editar código → probar en local con `npm start` → confirmar que funciona → `git add . && git commit -m "..." && git push` → Vercel despliega automático → revisar pestaña "Deployments" en vercel.com para confirmar que quedó "Ready" (no "Error").
